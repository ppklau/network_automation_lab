## The hierarchy

The ACME SoT is hierarchical. Lower layers inherit from higher layers and override only what is specific to that scope.

```
sot/global/          ← applies to all devices, all sites
sot/regions/         ← applies to all devices in a region
sot/sites/           ← applies to all devices at a site
sot/compliance/      ← zone policies, regulatory constraints
sot/bgp/             ← BGP route policies and community strings
sot/devices/         ← individual device records
```

When rendering a device config, the playbook loads global defaults first, then region, then site, then device — each layer overriding the previous one. A device file only needs to specify what is unique to that device.

## Global configuration

### `sot/global/platform_defaults.yml`

This file contains the values that should be consistent across all devices of a given platform type. Open it:

```bash
cat sot/global/platform_defaults.yml
```

You will find BGP timer defaults, NTP authentication settings, SSH configuration, AAA templates, and SNMP parameters. When ACME decides to change the BGP hold-timer from 90 seconds to 30 seconds, this file is the only place that changes. All device configs regenerate on the next pipeline run.

### `sot/global/ipam.yml`

The IPAM hierarchy defines the supernet allocation for each region and the subdivision within each DC:

```yaml
ipam:
  supernet: 10.0.0.0/8
  regions:
    emea_lon:     10.1.0.0/16
    americas_nyc: 10.10.0.0/16
    apac_sin:     10.20.0.0/16
    eu_fra:       10.30.0.0/16
  zones_per_dc:
    trading:   /24    # e.g., 10.1.1.0/24 at LON DC1
    corporate: /22    # e.g., 10.1.2.0/22 at LON DC1
    dmz:       /24    # e.g., 10.1.6.0/24 at LON DC1
    mgmt:      /25    # e.g., 10.1.0.0/25 at LON DC1
  branch_pool: 10.100.0.0/14
```

The branch pool (`10.100.0.0/14`) is subdivided into `/29` allocations per branch office. Each `/29` provides 6 usable host addresses — enough for a branch router, a management host, and a few workstations on a CORPORATE-only access VLAN. The `validate_sot.py` script checks that no two branches share a subnet from this pool.

## Compliance configuration

### `sot/compliance/zone_policies.yml`

This file encodes the zone access matrix — which zones can reach which other zones, and what policy applies at the boundary.

```bash
cat sot/compliance/zone_policies.yml
```

```yaml
zone_policies:
  - id: ZONE-001
    name: TRADING_isolation
    description: "TRADING zone must not be reachable from CORPORATE zone"
    requirement_refs: [REQ-007, REQ-008]
    intent_refs: [INTENT-001]
    enforcement: firewall_acl + vrf_separation
    audit_check: true

  - id: ZONE-002
    name: DMZ_access
    description: "TRADING may reach DMZ on TCP 443/8443/6000 only"
    requirement_refs: [REQ-008]
    intent_refs: [INTENT-002]
    enforcement: firewall_acl
    permitted_ports: [443, 8443, 6000]
    audit_check: true
```

Each zone policy has a unique ID, links to the business requirements that mandate it, links to the design intent that implements it, and specifies how it is enforced. The `audit_check: true` flag means the compliance report playbook will actively verify this control.

### `sot/compliance/regulatory.yml`

This file maps regulatory obligations to the zones and sites they apply to:

```yaml
regulatory_constraints:
  - region: eu_fra
    regulation: GDPR
    article: "Article 5(1)(c) — data minimisation"
    constraint: trading_zone_prohibited
    sites: [fra-dc1]

  - region: emea_lon
    regulation: MiFID_II
    article: "Article 48 — algorithmic trading"
    constraint: trading_must_be_isolated
    sites: [lon-dc1, lon-dc2]
```

> 🔵 **Strategic** — This file is what turns "we comply with GDPR" from a statement into a verifiable claim. The constraint `trading_zone_prohibited` in Frankfurt is:
> 1. Encoded in `sot/sites/fra-dc1.yml` as `vlans_prohibited: [100]`
> 2. Enforced by the Jinja2 template that skips VLAN 100 on Frankfurt devices
> 3. Verified by Batfish assertions in `test_frankfurt_isolation.py`
> 4. Reported in the compliance report generated by `playbooks/compliance_report.yml`
>
> A regulator asking "how do you ensure GDPR Article 5 data minimisation in your Frankfurt DC?" gets a pipeline run ID and a Batfish test report, not a conversation about what the team thinks they configured.

## BGP configuration

### `sot/bgp/route_policies.yml`

Global BGP route policies that apply to multiple devices:

```yaml
route_policies:
  - name: RM_INTERDC_SUMMARISE
    type: route_map
    description: "Inter-DC WAN — permit /16 aggregates only"
    intent_refs: [INTENT-006]
    entries:
      - seq: 10
        action: permit
        match: "ip address prefix-list PL_WAN_AGGREGATES"
      - seq: 100
        action: deny

  - name: RM_BRANCH_IN
    type: route_map
    description: "Accept only branch /29 from branch peers"
    intent_refs: [INTENT-005]
    entries:
      - seq: 10
        action: permit
        match: "ip address prefix-list PL_BRANCH_PREFIX"
      - seq: 100
        action: deny
```

Route policies are defined here and referenced in device files by name. When a device's BGP config references `export_policy: RM_INTERDC_SUMMARISE`, the template renders the full route-map from this definition. The route-map logic lives in one place; references to it are scattered across device files.

## Device files

### What a complete device file looks like

Read the full spine device file:

```bash
cat sot/devices/lon-dc1/spine-lon-01.yml
```

Work through each section:

**Identity:**
```yaml
hostname: spine-lon-01
platform: arista_eos
role: spine
site: lon-dc1
lab_state: active
```

`lab_state: active` means this device is instantiated in the containerlab topology and receives Ansible pushes. Devices with `lab_state: sot_only` are defined in the SoT (so all automation tooling knows about them) but are not running containers. London DC2, Singapore, Hong Kong, and most branches are `sot_only`.

**Loopback:**
```yaml
loopback:
  ip: 10.1.255.1/32
  description: "Spine-lon-01 router-ID and iBGP update-source"
```

The loopback is used as the BGP router-ID and update-source. All iBGP sessions in the London fabric use loopback IPs, so a single link failure does not drop a session.

**BGP (route reflector):**
```yaml
bgp:
  local_as: 65001
  router_id: 10.1.255.1
  role: route_reflector
  neighbors:
    - peer: 10.1.255.11
      remote_as: 65001
      description: "RR client — leaf-lon-01"
      route_reflector_client: true
      md5_password_ref: ibgp_spine01_leaf01
    ...
    - peer: 10.1.255.2
      remote_as: 65001
      description: "Inter-RR — spine-lon-02"
      route_reflector_client: false
      md5_password_ref: ibgp_spine01_spine02
```

The spines are route reflectors. Every leaf and the border router are RR clients. The two spines peer with each other as non-RR-client iBGP peers (the inter-RR session that prevents split-horizon black holes).

### Device files for `sot_only` nodes

Compare a `sot_only` device:

```bash
cat sot/devices/sin-dc1/border-sin-01.yml
```

This device has full BGP configuration, interface definitions, and compliance tags — but `lab_state: sot_only`. The pipeline's `generate_inventory.py` script excludes it from the Ansible inventory. Batfish still knows about it (the full SoT is loaded into Batfish snapshots). The compliance report will note that it is not actively checked.

## The validation layer

After reading the SoT, run the validation script to see what it checks:

```bash
python3 scripts/validate_sot.py
```

```
[OK] 75 device files parsed
[OK] No duplicate ASNs detected
[OK] No duplicate IP addresses detected
[OK] All intent_refs reference valid intent IDs
[OK] Frankfurt devices: VLAN 100 absent from all 6 devices
[OK] Branch prefix assignments: no overlaps detected
[OK] All md5_password_refs present in vault.yml
SoT validation passed — 75 devices, 0 errors
```

Now try introducing an error. Add `100` to the `vlans_prohibited` bypass by editing a Frankfurt device:

```bash
# Edit sot/devices/fra-dc1/leaf-fra-01.yml
# Find the vlans: section and add vlan_id: 100
```

Re-run the validator:

```
[FAIL] fra-dc1: leaf-fra-01 references VLAN 100 — prohibited at this site (vlans_prohibited)
SoT validation FAILED — 1 error
```

Restore the file (`git checkout sot/devices/fra-dc1/leaf-fra-01.yml`) and verify the validator passes again.

This is what the pipeline's validate stage does on every commit. Schema errors are caught before rendering, before Batfish, before anything reaches a device.

## What a complete branch onboarding actually requires

The entry you added above is the minimum the validator accepts. Onboarding a real branch requires touching six places in the SoT. Working through them manually once gives you a mental map of what the automation in Chapter 35 replaces.

### 1. Reserve the ASN — `sot/global/asn_pools.yml`

Check the correct regional range before picking an ASN. UK branches must come from `65100–65111`. The validator rejects duplicates, but it will not tell you which ASN to use — that is your responsibility.

```bash
cat sot/global/asn_pools.yml
```

### 2. Reserve the WAN prefix — `sot/global/ipam.yml`

Each branch gets a `/29` from the regional branch pool (`10.100.0.0/14` for EMEA). Find the next unallocated block and record it. The validator checks for overlaps across all branch files.

```bash
cat sot/global/ipam.yml
```

Look for the `branch_allocations` section and find the last used prefix in the UK range.

### 3. Add the branch entry — `sot/devices/branches/uk-branches.yml`

This is what you did in the exercise above. The required fields are: `hostname`, `asn`, `site_prefix`, `region`, `zone_permissions`, `compliance_tags`, `lab_state`.

### 4. Add the BGP MD5 password reference — `inventory/group_vars/vault.yml`

Every BGP session uses an MD5 password stored in the vault. Add an entry for the new branch:

```yaml
bgp_md5_branch_lon_05: CHANGEME
```

The `md5_password_ref` in the branch entry must match this key exactly. The validator enforces that no inline passwords appear in device files (INTENT-005).

### 5. Add the eBGP neighbour to the border router — `sot/devices/lon-dc1/border-lon-01.yml`

The branch router peers with `border-lon-01`. Open that device file and add a neighbour entry under `bgp.neighbors`:

```yaml
- peer_ip: 10.100.4.2        # border-lon-01 side of the /29
  remote_as: 65104
  description: "eBGP -> branch-lon-05 (Canary Wharf)"
  md5_password_ref: bgp_md5_branch_lon_05
  route_map_in: RM_BRANCH_IN
  route_map_out: RM_INTERDC_LON_NYC_IN
```

### 6. If lab_state: active — add a startup config

For a device that will be instantiated in containerlab, add an FRR startup config directory under `topology/startup_configs/<hostname>/` with `frr.conf` and `daemons`. Use `topology/startup_configs/branch-lon-01/` as a template.

### The manual process in practice

Six files. Three of them require cross-referencing each other (ASN range, IP range, vault key name). One error — a duplicate ASN, a typo in `md5_password_ref`, a prefix outside the pool — fails the validator or, worse, causes a BGP session to never come up.

This is the process that Chapter 35 automates. `bulk_import.py` takes a CSV row and handles steps 1–4 in one pass, with schema validation at the import gate. The border router neighbour update (step 5) is handled by re-rendering `border-lon-01` from the updated SoT.

For now, knowing what the automation replaces makes the later exercise meaningful rather than mechanical.

## Debrief

**What was practised:** Adding a new branch office to the SoT, running the validator, and fixing the errors it caught — without pushing anything to a device.

**Why it matters:** The SoT hierarchy (global → region → site → device) is not just file organisation — it encodes inheritance. A branch office inherits its zone permissions, compliance tags, and BGP policies from the regional and global defaults. Understanding this hierarchy means understanding why a device gets the configuration it gets.

**In production:** Most SoT-related incidents are not data errors — they are structural misunderstandings. An engineer adds a device file without realising that the regional defaults already set a constraint. The validator catches the symptoms, but understanding the hierarchy prevents the mistakes.
