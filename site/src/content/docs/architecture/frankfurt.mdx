<Aside type="caution" title="Regulatory constraint">
  Frankfurt has **no TRADING zone**. GDPR and BaFin rules require that EU-origin data cannot transit non-EU routing paths. Frankfurt peers only with London — never directly to New York or APAC. The Batfish pipeline enforces both constraints automatically.
</Aside>

## What the pipeline enforces

The Frankfurt compliance constraints are not comments in a document — they are Batfish assertions that run on every pipeline execution:

| Assertion | What it checks |
|-----------|---------------|
| `assert_no_trading_zone_fra` | No TRADING VLAN (100) on any FRA device |
| `assert_fra_peers_london_only` | FRA border has no BGP peers outside AS 65001 |
| `assert_no_direct_fra_nyc_path` | No route exists from FRA to NYC without transiting LON |
| `assert_fra_data_residency_tag` | All FRA SoT records carry `data_residency: eu` |

If any assertion fails, the pipeline blocks the merge. The change does not reach a device.

## IPAM

```yaml
eu_fra:
  supernet:   10.30.0.0/16
  corporate:  10.30.2.0/22            # No trading subnet
  dmz:        10.30.3.0/24
  mgmt:       10.30.0.128/25
  # 10.30.1.0/24 intentionally unallocated — no TRADING zone
```

The absence of `10.30.1.0/24` (which would be the TRADING subnet by convention) is itself a compliance artefact. Schema validation catches any attempt to add it.
