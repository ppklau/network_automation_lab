Network automation platforms share a common set of functional building blocks regardless of which products fill each role. This page presents that functional model first, then maps the specific tools used in this lab onto it.

<Aside type="note">
  Automation tooling is highly opinionated. The stack shown here is one valid approach — the same functions could be served by NetBox, Nautobot, Nornir, NSO, Terraform, Crossplane, Prometheus alternatives, or any number of other tools. What matters is that each function is covered, not which product covers it.
</Aside>

## Functional model

### What each function does

| Function | Responsibility |
|---|---|
| **Source of Truth** | Single authoritative record of what the network *should* look like. All config generation, validation, and drift detection derive from this. |
| **CI/CD Pipeline** | Validates changes, runs tests, sequences the delivery of config to devices. Enforces policy as code. |
| **Config Generation** | Renders device-specific configuration from templates and SoT data. Keeps config DRY and consistent. |
| **Workflow Orchestration** | Provides human approval gates, scheduling, multi-step runbooks, and a tamper-evident audit trail. |
| **Drift Detection** | Continuously compares the running state of devices against SoT intent. Surfaces unauthorised or unintended changes. |
| **Monitoring** | Collects operational metrics, interface state, routing protocol health, and any other telemetry the platform surfaces. |
| **Alerting** | Evaluates collected data against thresholds and rules. Notifies operators with troubleshooting bundle or triggers automated response. |
| **Auto-Remediation** | Drives corrective action in response to alerts or drift — re-applying config, escalating, or opening tickets — without requiring a human to initiate it. |

### Lab tool mapping

| Function | Lab Tool | Notes |
|---|---|---|
| **Source of Truth** | YAML files (Git) | Structured YAML in `sot/`, validated by JSONSchema. Version-controlled in GitLab. |
| **CI/CD Pipeline** | GitLab CI | `.gitlab-ci.yml` defines stages: validate → generate → verify → push. |
| **Config Generation** | Ansible + Jinja2 | Templates in `templates/`, rendered by Ansible and pushed via `playbooks/`. |
| **Workflow Orchestration** | Rundeck | Approval workflows, scheduled jobs, and audit logging for Day-2 operations. |
| **Drift Detection** | Ansible + Batfish | Ansible polls devices; Batfish models the topology and compares against intent. |
| **Monitoring** | Prometheus | Scrapes cEOS and FRR devices via SNMP and gNMI exporters. |
| **Alerting** | Grafana | Alert rules evaluate Prometheus metrics; notifications fire on threshold breaches. |
| **Auto-Remediation** | Rundeck + Ansible | Grafana webhooks trigger Rundeck jobs that run corrective Ansible playbooks. |

<Aside type="tip">
  Every tool in this stack has alternatives. NetBox or Nautobot could replace the YAML SoT. Jenkins or GitHub Actions could replace GitLab CI. Nornir could replace Ansible. NSO or Crossplane could replace the config generation layer. Alertmanager could route directly without Grafana. The architecture is the pattern — the tools are a choice.
</Aside>
